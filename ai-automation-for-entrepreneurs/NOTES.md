# Notes & Preferences

Working notes for designing and teaching this course. Update as preferences emerge.

## About the audience (instructor's working model)
- Non-technical entrepreneurs / small-business owners and solo founders.
- Motivated by time saved and money saved, not by the technology itself.
- Low tolerance for jargon. Every concept must earn its place with a concrete payoff.
- Likely to abandon if a lesson feels abstract or takes more than ~15 min.

## Design preferences
- **Mission-grounded:** every lesson ties to "automate a real workflow in YOUR business."
- **One tangible win per lesson.** No lesson is theory-only.
- **Show, then do.** Teach the minimum knowledge, then an immediate hands-on rep with feedback.
- **Cite real sources.** Never present parametric claims as fact — link to RESOURCES.md entries.
- **Beautiful, printable lessons** (Tufte-style): clean type, generous margins, works on paper.

## Hybrid delivery mechanics
- Self-paced HTML lessons released per week (this workspace's `./lessons/`).
- One live session per week: Q&A + guided build of that week's automation.
- Learners bring a real workflow from their own business as the running example.

## Building half — design principles (from research)
- **Design against the "comprehension debt" trap.** The known failure mode: learners generate 80% of an app instantly, then get stuck in the final 20%, burning credits in debug loops, owning code they can't read. Module 5 exists to counter exactly this — teach *enough* code reading + debugging + Git to get unstuck, no more.
- **No "blank chat box."** Teach learners to build *results and buttons* ("Draft this lease", "Audit this permit"), not chat interfaces. Users want results, not a text box.
- **One narrow job + a human review step.** Every first build should draft/sort/suggest while a person approves — same human-in-the-loop principle as the automation half.
- **Knowing your limits is empowerment, not failure.** Part of the mission is recognizing when a build is good enough vs. when to bring in a real developer.

## Tool anchors (recommended)
- **Automate:** Zapier (breadth + plain-English UI); Make as the visual alternative.
- **Build (vibe):** Lovable as the default starting point (most polished, least intimidating chat); Bolt for fast prototypes; v0 for UI; Replit to "graduate" into a real editor.
- **Avoid for this audience:** Cursor (requires coding + terminal comfort) — mention as a "next step," don't teach it.
